# PR 본문

## 작업 개요

CyphenEngine의 기존 동작을 새로운 구조 위에 동등하게 재정립하고, 기반 기능을 확장했습니다.

이번 브랜치는 새로운 상위 기능을 추가하기보다 기존 코드의 책임을 분리하고, 플랫폼 종속 구현과 엔진 공통 기능 사이의 경계를 명확히 하는 데 목적이 있습니다.

Renderer 연결과 애플리케이션 빌드 검증은 리팩토링 범위를 벗어나므로 후속 브랜치로 분리합니다.

## 주요 변경 사항

### 1. 프로젝트 및 빌드 구조 정리

- 현재 Source 트리를 Build / Core / Engine / HAL / Modules / Platform / Test 순으로 정리했습니다.
- 기존 Runtime의 File / Path / Time은 Editor와 Runtime이 함께 사용하는 기반 기능이므로 Core 계층으로 이동했습니다.
- Runtime과 Editor 계층은 전용 기능이 생기는 시점에 추가할 예정입니다.
- 플랫폼 정의를 `PlatformDefine`으로 분리했습니다.
- 빌드 대상과 설정 정보를 `BuildInfo`로 정리했습니다.
- 공통 헤더와 프로젝트 파일 구성을 현재 디렉터리 구조에 맞게 갱신했습니다.
- 소스 및 DevLog의 인코딩·줄바꿈 규칙을 정리했습니다.

### 2. 문자열 정책 정리

- 엔진 문자 타입을 `CChar`로 분리했습니다.
- `CString`과 기본 자료형 정책을 Core 계층으로 이동했습니다.
- 텍스트 인코딩 변환을 `TextCodec`으로 분리했습니다.
- UTF-8, UTF-8 BOM, UTF-16, ANSI 변환과 줄바꿈 정책을 명시했습니다.
- Windows API 문자열 변환은 `WindowsString`이 담당하도록 경계를 분리했습니다.

### 3. 애플리케이션 호스트 및 실행 구조 정리

- 기존 Launch 계층의 엔진 호스트를 Engine 계층으로 재배치했습니다.
- `CyphenEngine`을 플랫폼 독립 애플리케이션 호스트로 정리했습니다.
- 엔진 상태 전환과 종료 요청을 원자적으로 처리하도록 구성했습니다.
- Windows 메시지 루프와 엔진 실행 루프를 분리했습니다.
- 엔진 루프를 별도 스레드에서 실행하도록 변경했습니다.

### 4. Time 계층 분리

- 플랫폼 시간 조회를 `PlatformTime` HAL로 분리했습니다.
- Core `Time`은 단조 시간 기준의 경과 시간과 DeltaTime 계산을 담당합니다.
- 로컬 및 UTC 날짜·시각 조회를 구조화된 값으로 제공합니다.
- FPS는 Time의 책임이 아닌 상위 통계·프레임 계층의 파생 정보로 분리했습니다.

### 5. Path 및 구분자 정책 정리

- `Path`를 순수 경로 문자열 유틸리티로 재구성했습니다.
- Normalize, Combine, 디렉터리명, 파일명, 확장자 추출 기능을 제공합니다.
- 문자열 구분자 정책을 `Separator`와 `Separators`로 분리했습니다.
- Windows 및 Unix 구분자를 엔진 경로 정책에 맞게 처리합니다.
- 프로젝트 루트와 Content / Saved / Logs 등의 정책 경로는 Path에서 제거했습니다.
- 정책 경로는 향후 Application 또는 프로젝트 컨텍스트 계층에서 다룰 예정입니다.

### 6. File I/O 계층 재구성

- `File`을 파일 전체 내용을 처리하는 동기 I/O helper로 정리했습니다.
- 바이트 및 텍스트 읽기·덮어쓰기 API를 구성했습니다.
- 명시적인 바이트 및 텍스트 append API를 추가했습니다.
- 텍스트 인코딩과 BOM, 줄바꿈 처리 기준을 명확히 했습니다.
- 기본 저장 모델은 whole-file overwrite로 유지합니다.
- append는 로그·텔레메트리처럼 누적 기록이 필요한 경우 사용하는 명시적 기능으로 제한합니다.

### 7. FileSystem 관리 API 추가

다음 파일 시스템 관리 기능을 `FileSystem`으로 분리했습니다.

- 파일 및 디렉터리 존재 여부 확인
- 파일 생성·삭제·크기 조회
- 파일 복사·이동
- 단일 디렉터리 생성·삭제
- 디렉터리 트리 생성
- 비어 있지 않은 디렉터리 재귀 삭제
- 복사·이동 시 기존 대상 교체 여부를 명시하는 오버로드

`File`과 `FileSystem`은 서로를 소비하지 않는 Core의 형제 계층으로 유지합니다.

두 계층의 실제 플랫폼 파일 접근은 private `PlatformFile` HAL을 통해 수행합니다.

### 8. Windows PlatformFile 구현

- Win32 파일 API 호출을 Platform 계층 내부로 이동했습니다.
- 파일 읽기·쓰기·추가·생성·삭제·크기 조회를 구현했습니다.
- 파일 복사와 이동을 구현했습니다.
- 디렉터리 생성과 재귀 삭제를 구현했습니다.
- 엔진 경로를 Windows 경로 및 API 문자열로 변환하는 경계를 분리했습니다.

### 9. 내부 Core I/O 테스트 추가

Debug 빌드에서 실행되는 내부 테스트 계층을 `Source/Test/CoreIO`에 추가했습니다.

검증 범위:

- Path 정규화와 경로 요소 추출
- TextCodec 인코딩과 디코딩
- BOM 및 줄바꿈 정책
- File 바이트·텍스트 읽기와 쓰기
- File append
- FileSystem 파일 및 디렉터리 관리
- 복사·이동의 대상 교체 정책
- 디렉터리 트리 생성과 재귀 삭제
- Separator 정책

기록된 최종 결과:

```text
PASS=69
FAIL=0
```

## 리팩토링 결과

기존 `main`의 핵심 동작을 현재 구조 위에 동등하게 재정립하고, File I/O와 경로 처리 기능을 확장했습니다.

File I/O와 Path는 기존 기능을 단순히 이동하는 데 그치지 않고, 공개 Core API와 플랫폼별 구현의 책임을 분리했습니다.

엔진 실행 구조와 Time 역시 플랫폼 코드에서 분리하여 후속 기능을 연결할 수 있는 기반을 마련했습니다.

## Renderer 변경 범위

이번 diff에는 재배치된 타입 및 include 경로와의 정합성을 맞추기 위한 Renderer 관련 파일 변경이 포함됩니다.

다만 Renderer 초기화 연결과 실제 렌더링 기능 구성은 이번 브랜치의 범위에 포함하지 않습니다.

Renderer 모듈 연결과 Dx11 초기화는 후속 브랜치에서 진행합니다.

## 이번 브랜치에서 제외한 범위

다음 항목은 의도적으로 후속 작업으로 분리합니다.

- Renderer 모듈 연결 및 Dx11 초기화
- 애플리케이션 빌드·통합 테스트
- Logger / Diagnostics 재구성
- ResourceManager 및 비동기 리소스 로딩
- 공개 FileHandle / FileStream
- Seek / Tell 및 부분 읽기·쓰기
- 공개 디렉터리 열거 API
- timestamp 및 permission API
- 프로젝트 정책 경로 계층
- 기본 외부 텍스트 인코딩의 엔진 설정 연동

## 후속 브랜치

- 브랜치 2: Renderer 모듈 연결 및 렌더링 기반 구성
- 브랜치 3: 애플리케이션 빌드 및 엔진 통합 검증

## 검토 중점

- Core와 Platform 사이의 책임이 적절하게 분리됐는지
- File과 FileSystem의 공개 API 경계가 명확한지
- 문자열·인코딩·경로 정책이 플랫폼 구현과 분리됐는지
- 기존 동작이 새로운 구조 위에서 동등하게 재정립됐는지
- 후속 Renderer 및 애플리케이션 작업을 진행할 기반이 충분한지
