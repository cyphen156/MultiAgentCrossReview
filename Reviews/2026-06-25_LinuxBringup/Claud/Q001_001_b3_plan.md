# Q001_001 — #3(B3) Linux Bringup 전체 플랜

## 0. 출발 상태 (P2가 #3에 넘긴 것)

LinuxBuildPrep 결론: #2는 "컴파일된다"로 마감, **링크 미해결 심볼이 #3 시작점**.
미러 검증으로 확인한 사실:

- Linux Platform 구현 4종은 **빈 플레이스홀더**.
  - `Platform/Linux/Private/Launch.cpp` = `#include "pch.h"` 1줄
  - `ModuleLoader.cpp` / `PlatformFile.cpp` / `PlatformTime.cpp` = 0줄
  - → A는 "완성"이 아니라 **실구현**. Windows 동명 구현이 시그니처·동작 레퍼런스.
- HAL 계약 헤더는 확정 상태(아래 §3). Linux는 **동일 시그니처 concrete**만 채우면 됨(virtual 없음 — `build-abstraction-no-interfaces`).
- CMake는 Linux 전용 빌드 경로로 확정(`cmake-linux-only-role`), 드래프트 = `LinuxBuildPrep/Claud/artifacts/CMakeLists.txt`, Windows=MSVC vcxproj 유지.

## 1. 워크스트림 분해

| ID | 워크스트림 | 산출물(파일) | 외부 의존 | 선행 |
|----|-----------|------------|----------|------|
| **A1** | PlatformTime | `Platform/Linux/Private/PlatformTime.cpp` | `clock_gettime`, `localtime_r`/`gmtime_r` | 없음 |

| **B** | Launch / main | `Platform/Linux/Private/Launch.cpp` + Linux 진입점 | pthread(`-lpthread`) | A1~A3 |
| **C** | Windows-leaf 격리 마무리 | CMake 소스 분리 잔여 | — | 병행 가능 |

## 2. 권장 실행 순서 + 단계별 합격조건

순서 원칙: **링크 게이트를 먼저 닫는다**(P2가 정의한 #3 경계 해소) → 그 다음 표시(first-light) → 통합 테스트.

1. **A1 PlatformTime** — 가장 단순(함수 4개), 외부 라이브러리 0. 워밍업·툴체인 점검 겸용.
   - 합격: Linux에서 `Time` 경유 단조시간·Local/UTC 획득 동작, 관련 단위 테스트 그린.
2. **A2 PlatformFile** — 표면 최대(~16함수), CoreIo의 Linux 토대.
   - 합격: `CoreIoTests` Linux에서 그린(Windows 패리티). raw byte I/O + 디렉터리 + Copy/Move 전부.
3. **A3 ModuleLoader** — dlopen 3종 + 논리이름→`lib<name>.so` 변환.
   - 합격: `ModuleTests` Linux에서 그린, 모듈 DLL(.so) 로드/심볼/언로드.
4. **B Launch/main** — 진입점 + 생명주기.
   - **★ 링크 게이트**: `cmake` 빌드가 **링크까지 통과**(미해결 심볼 0). = #2가 남긴 경계의 공식 해소.
5. **C Windows-leaf 격리 마무리** — A~B 진행 중 발견되는 누수 즉시 봉합(병행).
   - 합격: Linux 빌드에 Windows-only 심볼(DX11/WindowsJpegCodec/.rc) 0 유입.
6. **D Renderer first-light** — OpenGL ES/EGL 백엔드(`renderer-backend-strategy`: Linux=GLES/fallback) + Linux JPEG leaf.
   - 합격: Linux에서 창 + TexturedQuad **표시**(#2_6 Profile 패리티의 Linux판 first light).
7. **E 통합 테스트** — 전 계층 Linux 실행 검증.
   - 합격: 통합 테스트 그린 → **#3 종료 조건 충족**.

## 3. 구현 대상 계약 표면 (헤더 확정분, Linux concrete가 채울 시그니처)

- **ModuleLoader**(friend ModuleManager, 전부 private static):
  `Load(CString, void*&)` → dlopen, `FindSymbol(void*, const char*)` → dlsym, `Unload(void*&)` → dlclose.
  추가: 논리 모듈명 → Linux 파일명(`lib<name>.so`) 변환은 구현 내부 책임(헤더 책임 명시).
- **PlatformFile**(friend File/FileSystem): `ReadAllBytes/ReadHead/WriteAllBytes/AppendAllBytes`,
  `Create/GetSize/Exists/Remove`, `DirectoryExists/MakeDirectory/DeleteDirectory/DeleteDirectoryRecursively`,
  `Copy(...,canReplace)`/`Move(...,canReplace)`. 핸들은 헤더 비노출 — open/close는 구현 내부.
  반환 정책: 성공/실패 bool만(OS 오류 세부는 후속 Diagnostics 단계).
- **PlatformTime**(friend Time): `Init()`, `MonotonicSeconds()` → `CLOCK_MONOTONIC`,
  `LocalDateTime()`/`UtcDateTime()` → `CDateTime` 변환.
- **Launch**(HAL/Public): 엔진 생명주기 + 메인 루프 별도 스레드. Linux 진입점이 이를 구동.

## 4. 미결정 / 사용자 결정 분기점

플랜 확정 전에 짚어야 할 선택지(상세 질문은 후속 Callback/별도 Q로):

1. **Linux 빌드 환경**: WSL2 vs 실제 배포판(컨테이너/VM), 툴체인(gcc vs clang), GLES 가용성.
   → A1 착수 전 가장 먼저 고정 필요(전 단계의 검증 토대).
2. **Linux JPEG leaf**: libjpeg-turbo(번들) vs stb_image. `dependency-policy-os-first`상 OS 부재(Linux WIC 없음)이므로 번들 정당 — 둘 중 택1은 D 단계.
3. **WindowsJpegCodec 위치**(Content vs Platform/Windows): #3 Linux JPEG leaf 도입 시 자연 분기(Decision 명시) → D에서 확정.
4. **GLES 버전/EGL 컨텍스트 경로**: 표시 검증의 최소 first-light 범위.
