# #1 리팩토링 브랜치 회고

지난 개발기를 쓴 이후로 꽤 많은 시간이 흘렀다.

2026년 5월 27일부터 리팩토링을 시작했으니, 어느새 한 달이 조금 넘었다. 그간의 성과를 간략하게 자랑하자면, 엔진의 모듈화 설계를 어느 정도 실제 코드 위에 올려두는 데 성공했다.

물론 앞으로도 뜯어고쳐야 할 것은 한참 남아 있다. 그 이야기는 다음 글과 다다음 글에서도 조금 더 자세히 다루게 될 것이다.

이 글을 작성하는 현재 시점은 2026년 7월 2일이다. 지금까지 총 세 번의 큰 작업 브랜치가 있었고, 이전 개발기에서 말했던 #3의 최소 목표에 도달한 상태이기 때문에, 이제는 한 번쯤 정리할 때가 되었다고 판단했다.

이번 글은 세 브랜치 중 첫 번째인 **#1 리팩토링 브랜치**에 관한 글이다.

크게 다섯 가지를 개편했다.

- 문서·인코딩 규약 정리
- Build / Core / HAL / Platform / Runtime 계층 재정립
- 플랫폼 처리 책임을 Runtime 밖으로 분리
- 플랫폼 정의 계층 분리
- 문자열·타입 규약 확정

글로 써 놓으니 짧아 보이지만, 실제로는 엔진 바닥을 한 번 다 들어냈다가 다시 까는 작업에 가까웠다. 모듈 간 책임과 경계를 근본부터 나눠야 했기 때문에 스트레스도 꽤 컸다.

이 브랜치 자체는 화면에 새로 그려지는 것이 없다.

지금 화면에서 두 장의 jpg가 교차로 렌더링되는 것도 사실 다음 브랜치들의 결과물이지, 이 브랜치의 직접적인 성과는 아니다. 대신 그 아래 구조를 거의 다 갈아엎었다.

## 1. 인코딩과 문서 규약

제일 먼저 정리한 것은 인코딩 정책이었다.

기존 프로젝트는 소스가 EUC-KR(CP-949)로 저장돼 있었고, 줄바꿈도 CRLF였다. 지금까지는 Windows에서만 작업했으니 큰 문제가 없었지만, Linux 빌드까지 가려면 이 부분은 반드시 먼저 걷어내야 했다.

```text
EUC-KR(CP-949)  ->  UTF-8 (65001)
CRLF            ->  LF
```

여기서 한 번 크게 데인 적이 있다.

git이 Windows 환경에서 커밋할 때 `core.autocrlf` 설정으로 LF를 다시 CRLF로 바꿔 버렸고, 애써 LF로 맞춘 파일이 커밋 순간 되돌아갔다. 그래서 `core.autocrlf`를 꺼서 문서 규약을 고정했다.

커밋 메시지 규칙과 DevLog 구조도 이때 바꿨다.

```text
DevLog/yy.mm.dd.txt  ->  DevLog/yyyy/yy.mm.dd.txt
```

사소해 보이지만, 이런 규약을 초반에 못 박아 두지 않으면 나중에 파일이 수백 개로 늘었을 때 손을 못 댄다.

리팩토링의 시작이 화려한 구조 변경이 아니라 인코딩·줄바꿈 정리였다는 게, 지금 와서 보면 오히려 순서가 맞았다. 최근 진행한 MultiAgentCrossReview에서도 이때 만든 규칙 덕분에 커밋 메시지와 DevLog 정리의 상당 부분을 자동화할 수 있었다.

요즘은 내가 대화 내용을 요약해 주고, "오늘 변경한 것을 규칙에 맞춰 커밋 메시지로 정리해 와"라고 시키면 형식은 거의 맞춰 나온다. 나는 거기서 사실관계와 오류만 잡는 쪽으로 일이 꽤 줄었다.

## 2. 플랫폼 책임을 Runtime에서 빼내기

과거에 구현했던 File과 Path를 보면서, 공통 API처럼 보이는 코드 안에 플랫폼 처리가 직접 들어와 있다고 느꼈다. 이번 브랜치에서는 그 부분을 실제로 걷어냈다.

먼저 폴더부터 정리했다. 그동안 Common과 Utility로 어정쩡하게 나뉘어 있던 것들을 Core 하나로 통합했다.

그리고 File / Path / Time의 공개 API는 Core 쪽에 남기고, 플랫폼에 종속되거나 빌드 타겟별 구현이 필요한 부분은 HAL 성격의 내부 계약과 Platform별 concrete 구현으로 나눴다.

```text
Core/Public                  공개 API
HAL                          Core와 Platform 사이의 내부 계약
Platform/{TargetPlatform}    플랫폼별 동일 이름 구현
```

계약은 한 곳에 남고, 그 계약을 각 플랫폼 구현이 채운다. 어느 구현을 실제 빌드에 넣을지는 빌드 시스템이 결정한다.

이 결정은 #2 브랜치 후반과 #3 브랜치에서 특히 크게 빛을 봤다. Windows 빌드에서는 Windows 구현만, Linux 빌드에서는 Linux 구현만 들어가게 만들 수 있었기 때문이다.

당시에는 CyphenEngine 본체의 위치도 정리했다. 플랫폼별 진입점은 Platform 아래의 `Launch.cpp`로 통일하고, 공통 실행 본체는 플랫폼 코드와 분리했다.

```text
Platform/Windows/Private/Launch.cpp
Platform/Linux/Private/Launch.cpp
```

이전처럼 `LaunchWindows`, `LaunchLinux`가 흩어져 있는 것이 아니라, 각 플랫폼 폴더 안에 동일한 이름의 진입점이 놓이게 된 것이다.

실행 진입점은 플랫폼이 소유하고, 엔진 본체는 공통 코드로 내려간다. 이 경계가 이제 폴더 구조만 봐도 보인다.

## 3. 플랫폼 정의를 세 계층으로 나누기

`define.h`가 너무 많은 일을 하고 있다고 생각했다.

플랫폼 감지, 문자열 정책, 경로 구분자, 시간 타입, 로그 매크로, 싱글톤 매크로가 전부 한 파일에 들어 있었다. 이번에 이걸 세 계층으로 쪼갰다.

```text
PlatformDefine.h    빌드 시스템이 주입한 TARGET_PLATFORM_*을
                    엔진 내부 PLATFORM_* 기준으로 확정

framework.h         플랫폼별 시스템 헤더 포함
                    플랫폼 타입과 디버그 출력 경계 준비

define.h            플랫폼 분기를 제거한 공통 매크로
```

핵심은 빌드 시스템과 엔진 내부의 경계다.

빌드 쪽은 `TARGET_PLATFORM_WINDOWS` 같은 값을 주입하고, `PlatformDefine.h`가 그걸 받아 `PLATFORM_WINDOWS = 1` 같은 엔진 내부 기준으로 바꾼다. 엔진 코드는 이제 빌드 시스템이 뭘 어떻게 주입했는지 몰라도 되고, `PLATFORM_*` 하나만 보면 된다.

`define.h`는 이 과정에서 훨씬 가벼워졌다. 플랫폼 분기는 빠지고, 공통 매크로만 남는 방향으로 정리했다.

## 4. 타입은 using, 리터럴은 매크로

이 리팩토링 과정 중에 가장 오래 고민한 것 중 하나가 문자열이었다. 그리고 이 고민은 여전히 계속된다.

Windows는 wide 문자열을 쓰고, Linux는 narrow 문자열을 쓰려면 코드에 박히는 리터럴까지 플랫폼에 맞춰 바뀌어야 한다. 그런데 모든 것을 매크로로 만들면 코드가 너무 탁해진다.

이번에 원칙을 확실히 갈랐다.

**자료형은 using, 리터럴만 매크로.**

```cpp
using CString = std::wstring;   // Windows
using CString = std::string;    // Linux

using CChar = wchar_t;          // Windows
using CChar = char;             // Linux

CTEXT("감사합니다")             // Windows: L"감사합니다" / Linux: "감사합니다"
```

타입은 컴파일러가 이해하는 `using`으로 충분하다. 매크로로 만들 이유가 없다.

반대로 리터럴 앞의 `L`은 `using`으로 붙일 수 없다. 그래서 이 지점에만 `CTEXT` 매크로가 남는다.

딱 필요한 곳에만 매크로가 있는 구조다.

이름도 바꿨다.

```text
TString / TChar / TTEXT  ->  CString / CChar / CTEXT
```

여기서 `C`는 그냥 접두어가 아니라, 엔진이 플랫폼·빌드 차이를 흡수해 하나의 공통 규칙으로 확정한 타입이라는 표시다. Windows SDK의 `TCHAR`와 헷갈리지 않게, 그리고 이름 자체가 엔진의 정책을 담도록 골랐다.

## 5. PCH를 얇게, 그리고 빌드가 구현을 고른다

마지막으로 두 가지를 정리했다.

하나는 PCH다.

그동안 `pch.h`가 표준 라이브러리를 전부 끌어안고 있었는데, 이걸 걷어내고 각 구현 파일이 자기가 필요한 표준 헤더를 직접 include하는 원칙으로 바꿨다.

`pch.h`는 이제 전처리 환경 준비와 include 순서 고정만 담당한다.

```text
PlatformDefine.h  ->  framework.h  ->  define.h
```

이러면 빌드는 조금 느려질 수 있다.

하지만 각 파일이 무엇에 의존하는지가 파일 안에서 바로 보이고, "이게 어디서 딸려 온 헤더지?" 하는 문제가 줄어든다.

다른 하나는 "어느 cpp를 빌드에 넣을지"는 빌드 시스템이 정한다는 원칙이다.

```text
Windows 빌드    Platform/Windows/Private/*.cpp 포함, Linux 구현 제외
Linux 빌드      Platform/Linux/Private/*.cpp 포함, Windows 구현 제외
```

둘 다 넣으면 중복 정의가 나고, 하나도 안 넣으면 링크 에러가 난다.

Runtime 코드 안에서 `#if PLATFORM_WINDOWS`로 갈라 두면 당장은 편하다. 하지만 그렇게 하면 플랫폼 규약이 공통 코드로 새어 나온다.

빌드 타겟이 Windows면 Windows 구현만, Linux면 Linux 구현만 컴파일하는 게 자연스럽다고 판단했다.

## 정리하며

이번 브랜치의 결론을 추리면 이렇다.

- 인코딩·줄바꿈·문서 규약을 먼저 고정했다.
- Common·Utility를 Core로 통합하고, 계층 경계를 폴더 구조에 새겼다.
- File / Path / Time은 공개 API, 내부 계약, 플랫폼별 구현으로 나눴다.
- 플랫폼별 진입점은 각 Platform 아래의 `Launch.cpp`로 통일했다.
- 플랫폼 정의를 `PlatformDefine -> framework -> define` 세 계층으로 나눴다.
- 자료형은 `using`, 리터럴만 `CTEXT` 매크로로 정리했다.
- `T` 접두 문자열 규약을 `C` 접두 규약으로 바꿨다.
- `pch.h`를 얇게 만들고, 표준 헤더는 각 파일이 직접 포함하게 했다.
- 플랫폼별 cpp 선택은 빌드 시스템의 책임으로 넘겼다.

화면상 달라진 건 거의 없지만, 이제 엔진은 "Windows에서만 도는 코드"가 아니라 "플랫폼을 빌드 시점에 고를 수 있는 코드"가 됐다.

Linux 빌드로 넘어갈 수 있는 바닥이 이때 깔린 셈이다.

다음 글에서는 이 위에 올라간 첫 번째 실체 렌더러 모듈을 엔진에 붙인 이야기, 즉 #2 브랜치를 정리하려 한다.

모듈을 DLL로 떼어내고, 엔진이 그걸 로드해 화면에 무언가를 그리기까지의 과정이다.
